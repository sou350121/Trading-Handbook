> **核心定位**：落點於「文本另类 × Agent自主演进」軸，填補了開放領域金融代理缺乏端到端時效性搜索與多步推理評估基準的 prior gap。

**五軸座標**

| 數據模態 | 時間尺度 | 學習範式 | Alpha機制 | 人機協作 |
|:-:|:-:|:-:|:-:|:-:|
| `文本另类` | `跨周期` | `生成式大模型` | `因子挖掘` | `Agent自主演进` |

**Status:** v0.5 — 基於 QuantML 導讀 + 原論文（如有）。benchmark 細節待升 v1。
**TL;DR:** ①首個完全開源的金融搜索推理基準，涵蓋635題三檔任務，系統評測21款LLM代理的時效檢索與多步推理能力。②核心trick在於摒棄預置上下文，構建需調用搜索/API/SQL的端到端代理任務，並引入70位專家標注與多階段質檢。③這對「Agent自主演进」軸的意義在於，將評估從靜態RAG拉升至動態工具調度與證據調和，直接對齊高頻/事件驅動分析的工作流。④導讀給出Grok 4 (web)在全球子集得分68.9，逼近人類專家75.0基準。

**X-Ray.** FinSearchComp 將 LLM 評估從「閉卷記憶」推向「開卷工具調度」，實質上重構了文本另类因子挖掘的驗證層。舊工程坑在於傳統 FinQA/FinanceQA 預置上下文，掩蓋了代理在數據新鮮度管理、口徑對齊與多源衝突解決上的真實缺陷；本基準透過 T1-T3 遞進設計，強制暴露檢索深度不足與參數化記憶過時等失效模式。然而，其 envelope 仍受限於靜態問答範式：它衡量的是「單次查詢準確率」，而非因子在組合層面的風險調整收益或執行滑點。對量化讀者而言，此基準的價值不在於挑出得分最高的模型，而在於提供一套可重複的「工具調度壓力測試」協議。當 Agent 需自主演進以挖掘非結構化文本 alpha 時，FinSearchComp 的 T3 任務直接對應多跳推理與財務口徑標準化能力，是過濾「幻覺驅動型」代理的硬閾值。後續應將其與回測框架對接，驗證搜索增強後的因子在實盤中的信息係數衰減曲線。

## §1 · 架構 / Core Mechanism
| 維度 | 舊基準 (FinQA/BrowseComp等) | FinSearchComp | 解構意義 |
|---|---|---|---|
| 上下文供給 | 預置文檔片段 | 零預置，強制調用搜索/API/SQL | 消除「閉卷作弊」，暴露真實檢索依賴 |
| 任務結構 | 單步閱讀理解 / 孤立事實查找 | T1-T3 遞進（時效→歷史→多週期綜合） | 對齊分析師工作流，測評證據調和能力 |
| 評估焦點 | 靜態答案匹配 | 時效性驗證 + 口徑對齊 + 多源衝突解決 | 引入金融數據特有的新鮮度與合規約束 |

⚡ **Eureka:** 「不給答案，只給問題與工具權限，讓代理在真實數據流中自證推理鏈。」
**信息流:**
```
User Query -> Agent Planner -> [Search/API/SQL Router] -> External Data Stream -> Validator/Reasoner -> Final Answer
```

## §2 · 數學層
📌 **Napkin Formula:** 
$Score_{task} = \frac{N_{correct}}{N_{total}} \times 100$
複雜度隨任務階梯遞增：$C(T1) < C(T2) < C(T3)$，其中 $C(T3)$ 涉及多跳推理鏈長度 $L$ 與數據口徑對齊矩陣 $M_{align}$。
**直覺:** 將金融搜索轉化為離散決策路徑的正確率統計，T3 的複雜度不在於檢索量，而在於時間軸對齊與實體解析的組合爆炸。
**Loss/訓練細節:** 無模型訓練，純推理評測；依賴專家標注答案集與多階段質檢協議進行離散評分。

## §3 · 數據層
規模：635 題。市場：全球子集 + 大中華區子集。語言：英文 + 中文。來源：70 位金融專家設計，多階段質檢。頻率/時段：涵蓋實時（T1）、特定財報期（T2）、長週期歷史（T3）。樣本外假設：問題設計刻意避開模型預訓練截止日期的常見知識，強制依賴外部檢索；容量假設未披露，但問題設計偏向中低頻分析場景，不適用於高頻 tick 級驗證。

## §5 · 評測 / Benchmark
| 數據集/任務 | Metric | 前SOTA/基線 | 本方法/評測對象 | Δ |
|---|---|---|---|---|
| 全球子集總體 | 準確率 | 人類專家 75.0 | Grok 4 (web) 68.9 | -6.1 |
| 大中華區子集總體 | 準確率 | 人類專家 88.3 | 豆包 DouBao (web) 54.2 | -34.1 |
| 全球子集 T1 | 準確率 | DeepSeek官網 37.3 | YuanBao(金融插件) 69.2 | +31.9 |
| 全球子集 T1 | 準確率 | 標準版模型 | 思考版模型 | -7.0 |

**解讀:** Δ 真 capability vs 過擬合/成本未計。T1 的 +31.9 與 -7.0 顯示工具調度質量與任務匹配度遠比「思考版」參數量關鍵；金融插件的增益來自數據源直連而非模型本質躍升。人類基準的 75.0/88.3 與代理的負 Δ 證明當前系統在複雜金融語境下仍存在顯著能力缺口，未達實盤部署閾值。T3 任務中僅少數模型得分未超過30，反映多步證據調和仍是瓶頸，非單純檢索能力可彌補。

## §6 · 失效與隱含假設
**6.1 論文自述 limitations:** 模型表現顯著落後人類；常見失敗原因為搜索深度不足與使用過時信息；T3 任務僅限數據點少於5個的簡單查詢能成功。
**6.2 推斷的隱含假設:** 
- **Regime 依賴:** 依賴公開市場數據可獲性，對非公開/私有數據或極端流動性枯竭行情失效。
- **容量/成本:** 未計入 API 調用成本與延遲，不適用於低延遲交易或大規模因子掃描。
- **數據泄漏:** 問題設計雖避開預訓練知識，但部分歷史數據可能已沉澱於模型參數中，存在隱性記憶作弊風險。
- **Survivorship:** 僅覆蓋主流上市公司與指數，未處理退市/粉單市場樣本偏差。

## §8 · For the Reader
- **因子研究員**：將 T3 任務映射至財務口徑標準化流程，用其測試自動化因子構建腳本在處理財報重述與單位換算時的魯棒性。
- **LLM-Agent 開發者**：不要盲目堆疊 RAG 檢索器，優先優化 Router 的時效性判斷模塊與多源衝突解決邏輯，參考 YuanBao 插件架構。
- **組合配置/風控**：將基準得分轉化為「代理決策置信度」權重，得分低於人類基準 70% 的模型僅限於信息彙總，嚴禁直接輸出交易信號。
