# 自動化研究迴圈建造指南：把 loop engineering 接上 trial accounting

> 這一族坐落在「研究流程」而非「方法族」的層級。它真正在賭的是一句話：**當你把因子/策略研究自動化，你買到的是搜索吞吐，付掉的是統計顯著性——而這筆帳如果不記，你會拿到一張漂亮的、通過所有檢驗的偽發現證書。**
>
> 這不是一篇綜述，是一份**建造指南**。它的每一條約束都能追溯到 [自動化搜索 vs 預註冊驗證](/crossing/autoresearch-vs-preregistration) 裡經對抗驗證的證據，或一份 [五案全結的實戰紀錄](/use-cases/philosophy-quantification)。

**Status:** v0.6 — Opus 手寫。標「**未驗證**」的部分是設計提案而非既成實踐；標「**已驗證**」的有原文出處。這份指南自己也還沒有樣本外紀錄——見文末「這份指南自己的 falsifiable 預測」。

## 為什麼需要這份指南

自動化研究迴圈（loop engineering）這幾年的論述——Steinberger 的「不要再 prompt，去設計會 prompt 你的 agent 的迴圈」、[loop-engineering](https://github.com/cobusgreyling/loop-engineering) 這類把迴圈拆成五個原語（排程 / worktree / skill / MCP 連接器 / 子 agent）加記憶的框架——解決的全是**工程**問題：怎麼讓 agent 持續跑、怎麼隔離、怎麼記住狀態、怎麼設人工閘門。

這些框架**沒有一個**處理研究迴圈特有的那個問題：**每跑一輪，你的統計顯著性就少一分。**

而量化研究側的文獻正好相反——它把多重檢定、deflated Sharpe、PBO/CSCV 講得很透，但完全不談怎麼把它接進一個 7×24 自動跑的系統。實測結果：對 AlphaGen / AlphaQCM / TLRS / RiskMiner / AlphaEval 全文檢索 `multiple testing`、`data snooping`、`deflated Sharpe`、`PBO`、`CSCV`、`false discovery`、`Bonferroni`，**命中次數全部為 0**。

這份指南就是那道接縫。

## 第一原則：那條不能跨越的邊界

自動化研究迴圈的架構由**一條禁令**決定，其餘都是它的推論。

Bailey / Borwein / López de Prado / Zhu 在提出 PBO/CSCV 的同一篇論文的「限制與誤用」一節寫道（**已驗證**，逐字）：

> 我們必須警告讀者不要用 CSCV 來**指導**最優策略的搜索。那將構成對本方法的嚴重誤用……當一個度量變成目標，它就不再是好的度量……PBO 不應該是選擇所依賴的目標函數。

同一節堵死了「那我用 holdout 就好」這條退路：

> 只要研究者試過不只一組配置，過擬合就永遠存在……hold-out 方法不考慮嘗試次數……「這個回測過擬合了嗎」的答案不是真或假，而是一個**取決於試驗次數的機率**。

推論：

1. **反過擬合指標不能出現在任何 fitness / 選擇 / 排序函數裡。** 它一旦可被優化就被優化掉。
2. **驗證層必須在迴圈物理上構不到的地方。** 不是「約定不用」，是**架構上不可達**。
3. **閘門的輸出不是 boolean。** 是條件於 N 的機率。一個只會回答「通過/不通過」的閘門，對 trial count 完全失明。

所以架構只有一種形態——**分層，且層間是單向的**：

```
┌─ 搜索層 ────────────────────────────────────────┐
│  全自動、任意吞吐、任意 fitness                    │
│  產出：候選 + Trial Ledger（含全部被丟棄的）        │
└──────────────┬──────────────────────────────────┘
               │ 只有 ledger 和候選規格往下走
               │ 模型/權重/搜索狀態不往下走
┌──────────────▼──────────────────────────────────┐
│  篩選層（自動）                                    │
│  正交化 / 已知因子吸收 / 成本與容量 / 擁擠度        │
│  ⚠ fitness 不得含任何反過擬合指標                  │
└──────────────┬──────────────────────────────────┘
               │ 人在這裡釘門柱，門柱先於計算提交
┌──────────────▼──────────────────────────────────┐
│  驗證層（不可再生資源）                            │
│  deflated Sharpe / PBO（用 effective N）           │
│  + 滾動起點 + 凍結 holdout + CPCV                  │
│  ⚠ 迴圈對這一層沒有寫入權，也沒有讀取回授           │
└──────────────┬──────────────────────────────────┘
               ▼
        輸出：P(這是真的 | N = 你實際試了幾次)
```

## Trial Ledger：唯一無法事後重建的東西

這是整套設計裡**最重要也最容易被跳過**的一件事。所有其他東西（回測引擎、統計檢定、報告）都能事後補；**trial count 不能**。你今天沒記，明天就永遠算不出 deflated Sharpe。

**必須登記被丟棄的候選。** 這是最反直覺的一條——直覺會說「都丟掉了記它幹嘛」。但多重性懲罰的分母就是**你看過的東西的總數**，不是你留下的東西的總數。一個在第 3 步就被砍掉的候選，一樣花掉了你的顯著性。

一個可用的最小 schema（**未驗證**，設計提案）：

| 欄位 | 為什麼要 |
|---|---|
| `trial_id` | 全域單調遞增，跨 run 不重置——重置等於偽造 N |
| `run_id` / `parent_trial_id` | 支持樹狀搜索的譜系；effective N 的聚類要用 |
| `spec_hash` | 候選規格的正規化雜湊——用來偵測「換皮同一個東西」 |
| `generated_by` | 哪個 agent / 哪個 fitness / 哪個 prompt 版本 |
| `evaluated_on` | 資料窗、宇宙、成本模型（**成本模型必須進 ledger**） |
| `outcome` | `kept` / `dropped` / `error`，以及丟棄理由 |
| `metrics` | 全部指標，不只入選的那一個 |
| `holdout_touched` | **布林 + 次數**——holdout 每被看一次就要留痕 |

> **`spec_hash` 是防「假多樣性」的關鍵。** 自動生成器很擅長產出語法不同、語義相同的候選。若不做正規化去重，你的名義 N 會虛高（讓你過度懲罰），而你的**有效**多樣性其實很低（讓你以為分散了其實沒有）。這兩個錯誤方向相反，但會同時發生。

## 前置閘門：在燒預算之前先證明有東西可挖

這條張力上成本最低的一道閘門，來自一篇 2007 年、19 年來幾乎沒人用的論文。Chen & Navet 的 **pretest**（**已驗證**）：在跑完整搜索之前，先讓你的 miner 跟兩個明確的 null 對打——

- **Null A：同等強度的隨機搜索。** 精確匹配你的 miner 會評估的唯一候選數（原文：500 族群 × 100 世代 ≈ 5 萬個隨機候選）。
- **Null B：零智能 / 樂透式交易。** 隨機進出場。

原文的停止規則（逐字）：

> 如果 pretest 的結果顯示 GP 沒有優於隨機搜索或隨機交易行為的統計證據，那我們認為就沒有必要再投入資源於 GP。

**它是兩分支的，不是硬停**——這個區分很重要：
- 輸給 **Null B（樂透）** → 這段資料裡沒有可學的東西 → **放棄這個問題**。
- 在可學的資料上輸給 **Null A（隨機搜索）** → 你的搜索機器配錯了 → 作者明確授權**繼續調參**。

**誠實揭露**：原作者自稱只是初步證據，3 個交易所有 1 個不一致，**19 年來沒有任何獨立複現**，現代 RL/LLM 挖掘文獻採用率接近零。它是「原理紮實但非標準」的協定。但它是唯一一個能在花錢之前否決整個計畫的設計，成本又幾乎為零——**不做它的理由只有一個：你不想知道答案。**

## 驗證層：三件事，一個未解

**1. Deflated Sharpe / PBO，用 effective N。**（**已驗證**方法，**未驗證**在小宇宙的可用性）

為什麼一定要 effective N 而不是名義 N：名義 N 高估多重性，因為候選之間高度相關。López de Prado & Lewis (2018) 用無監督聚類估計有效試驗數，Harvey & Liu 也有相應處理。

**這是這份指南最大的未解問題**，而且對小宇宙特別致命：在一個 20–30 檔 ETF 的宇宙裡，候選訊號幾乎是彼此的克隆，名義 N 與有效 N 的差距可能是**數量級**級別——而這個差距直接決定 deflated Sharpe 閘門到底能不能用。用名義 N 會嚴重過度懲罰（什麼都過不了），用有效 N 又需要一個本身未被驗證的聚類步驟。**本指南未評估。**

**2. CPCV 優於單路徑與 walk-forward。**（**已驗證**）Arian, Norouzi & Seco (2024, Knowledge-Based Systems 305) 的結論：單路徑與 walk-forward 方法「在偽發現防護上有顯著缺陷」，而組合式淨化交叉驗證（CPCV）在 PBO 與 deflated Sharpe 兩項上支配它們。

**3. 滾動起點是最便宜、最誠實的一關。**（**已驗證**於實戰側）[五案紀錄](/use-cases/philosophy-quantification) 裡有三案死在這一關（13F 1/6、Greenblatt 0/7、US 四引擎 4/7），而這三案在**單一固定窗**上都看起來不錯甚至很好。對照文獻側：AlphaGen 用 2020-2021 一個固定窗、AlphaQCM 用 2021-2022 一個固定窗——**兩者都不會照出這個東西**。而 AlphaQCM 自己的附錄 G.3 一旦把窗延長到 2024，IC 就從 8–9.5% 掉到 4–5.9%。

## 五個 loop 原語 → 研究迴圈的哪一節

把 loop engineering 的原語映射到研究迴圈，並標出每一個原語**在這個場景裡特有的坑**：

| 原語 | 在研究迴圈裡的角色 | 這個場景特有的坑 |
|---|---|---|
| **排程 / 自動化** | 候選生成的節奏 | 排程頻率直接決定 N 的增長率——**排程設定就是顯著性預算設定**，多數人沒意識到這一點 |
| **Worktree 隔離** | 每個候選在乾淨副本裡評估 | 隔離解決的是檔案衝突，**不解決統計污染**；兩者容易被混為一談 |
| **Skill / 持久知識** | 領域約束、成本模型、宇宙定義 | 把 gate 定義寫進 skill = 讓迴圈可以讀到門柱 → **門柱可被優化** → 破壞分層 |
| **MCP / 連接器** | 資料源、回測引擎 | 資料的 point-in-time 正確性是前提，不是細節（見 [五案](/use-cases/philosophy-quantification) 的隱性現金與 QFQ 混基底事故） |
| **子 agent（maker/checker）** | 生成 vs 驗證分離 | **checker 若與 maker 同模型同 prompt 家族，它不是獨立驗證**，是同一個先驗投了兩票 |
| **記憶 / 狀態** | Trial Ledger 的載體 | 狀態檔會被壓縮/輪替——**ledger 必須 append-only 且不可壓縮**，否則 N 會悄悄變小 |

> 最後一列的第三行是最隱蔽的一個坑：把閘門定義寫進 agent 的 skill 檔（這是 loop engineering 框架推薦的做法），會讓迴圈**看得到**門柱。看得到就會被優化到——不是因為 agent 作弊，是因為那就是優化的定義。門柱必須存在於迴圈的讀取範圍之外。

## 停止規則、預算與 kill switch

**停止規則**（**未驗證**，設計提案）。研究迴圈需要三種，缺一不可：

1. **前置停止**：pretest 未通過 → 不啟動。
2. **無進展停止**：連續 K 輪沒有新的、通過篩選層的候選 → 停，升級給人。這是 loop engineering 的 circuit breaker 在研究場景的形態。
3. **顯著性預算耗盡**：`N` 增長到讓 deflated Sharpe 門檻高於任何候選能達到的水準 → **停止搜索，因為再搜也不可能可信**。這一條在既有框架裡完全沒有對應物，是研究迴圈獨有的。

第 3 條值得停下來想一秒：它意味著**搜索預算的上限不是算力，也不是錢，是統計**。這與所有工程直覺相反。

**成本與 kill switch**：token 上限、單輪超時、以及一個人可按的停止開關。這是 loop engineering 框架做得最好的部分，直接採用即可。

**約束檔（denylist）**：明文寫死迴圈絕對不能碰的路徑與動作。對會動真錢的系統，這是唯一不可省的一份文件——實盤下單路徑、憑證、SSOT 權重檔、以及**驗證層的 holdout 資料**。把 holdout 寫進 denylist，是把「架構上不可達」從口號變成可執行的唯一辦法。

## Anti-patterns（每一條都有人做過）

- ❌ **把 PBO / deflated Sharpe 放進 fitness。** 原作者明文禁止。做了以後你的指標會很好看，而且你不會知道它壞了。
- ❌ **holdout 反覆使用。** 每偷看一次它就退化一分，退化**不可逆也不可觀測**。這是驗證層的頭號自殺方式。
- ❌ **只報 IC。** 整條文獻的通病。IC 沒有扣成本、沒有風險調整、也沒有多重性校正。最極端的一例：某 LLM agent 論文的 S&P 500 結果只收賣出費、買入免費，而那個 IR 1.05 建立在 IC 0.0056 上。
- ❌ **信任單次運行。** 種子離散度常大於效果量（實測：GP + 過濾在全市場 IC **0.84% ± 2.27%**）。任何沒有多種子的數字都是軼事。
- ❌ **在小截面宇宙上做橫截面因子挖掘。** 這一族的全部證據建立在 CSI 300/500 這種數百檔的寬度上。**20–30 檔 ETF 的截面 IC 幾乎不承載資訊**——這時候該搜索的是規則與配置，不是公式。
- ❌ **迭代到通過。** 改一個部件、重跑、沒過、再改——這是元過擬合，而且它連 trial ledger 都很難抓到，因為每次都「只改了一點」。實戰側的處置是：**消融顯示要砍的部件修不了那一關 → 不做 v2**。
- ❌ **把 loop readiness score 當安全指標。** `loop-audit` 這類工具量的是檔案存在性（有沒有 `STATE.md` / `LOOP.md` / `loop-budget.md`），不是統計效力，也不是實盤安全。分數高不代表你的發現是真的。

## 讀者分流

- **想從零建一條迴圈**：先做 pretest 和 trial ledger 這兩件事，其他都可以晚一點。它們是唯二無法事後補的。
- **已經有一條迴圈在跑**：先回答一個問題——**你到目前為止總共評估過幾個候選？** 答不出來，你的所有結論都沒有已知的顯著性。
- **在小宇宙（ETF 輪動 / 配置層）**：這一族的方法多半不能直接搬。你的搜索對象是規則與風險預算，不是公式；你的 N 天然較小（這是好事），但你的候選相關性極高（這是壞事），effective N 是你的核心未解問題。
- **在做 LLM agent 研究迴圈**：你的 checker 子 agent 若與 maker 同模型，它不是獨立驗證。考慮異質模型或明確的對抗性 prompt（要求 refute 而非 confirm）。

## 這份指南自己的 falsifiable 預測

按本手冊的規矩，把話說死並帶日期：

1. **到 2027-Q2 為止，不會出現任何一篇自動因子挖掘論文，同時公開（a）預註冊協定、（b）明確的累計 trial count、（c）deflated Sharpe 或 PBO 數值。** 若出現，本頁的核心論點（「這是整個次領域的結構缺口」）需要降級為「曾經是」。
2. **AlphaQCM 類方法在 2025-2026 的新樣本外窗上，IC 會繼續低於其 2021-2022 窗的水準**，除非採用未經測試的持續 refit。若不成立，則附錄 G.3 的衰減應被重新解釋為中國 A 股 2023-2024 的體制事件而非方法性衰減。
3. **在 20–30 檔標的的宇宙上，用名義 N 計算的 deflated Sharpe 閘門會拒絕掉全部候選**（過度懲罰），因而必須依賴 effective N 估計——而該估計本身將成為新的、未被驗證的自由度。**這是本指南最可能出錯的地方。**
